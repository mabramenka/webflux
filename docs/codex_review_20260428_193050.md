# Independent Architecture Review

Date: 2026-04-28

## 1. Executive Summary

Overall architecture rating: **Risky**

Top strengths:
- Clear aggregation pipeline: controller -> `AggregateService` -> part planner/executor -> workflow enrichers -> HTTP clients.
- Good modern Spring baseline: Java 21, Spring Boot 4.0.6, WebFlux, HTTP service clients, `ProblemDetail`, validation, Gradle checks.
- Strong test volume: 52 test classes; `./gradlew test` and `./gradlew build` both pass.

Top risks:
- The implemented contract treats missing enrichment keys and empty enrichment data as successful soft outcomes, but the stated business goal says enrichment data is required.
- Top-level enrichers tolerate missing downstream items and malformed-but-valid schemas as `EMPTY`/partial success.
- Public API/OpenAPI does not fully describe the real query/header/error contract, and a hidden `fields` query parameter can change main-service projections.

Fit for purpose: **not as-is**. The structure can be salvaged, but the success/error semantics must be corrected before adding features.

## 2. Architecture Map

Observed flow:
- Entry points: `AggregateController` exposes `POST /api/{apiVersion}/aggregate` and `GET /api/{apiVersion}/aggregate/{id}` in [AggregateController.java](../src/main/java/dev/abramenka/aggregation/api/AggregateController.java#L24).
- Request DTO: `AggregateRequest` validates `ids` and `include` with Jakarta Bean Validation in [AggregateRequest.java](../src/main/java/dev/abramenka/aggregation/api/AggregateRequest.java#L11).
- Application service: `AggregateService.aggregate(...)` plans parts, calls main `AccountGroups`, executes enrichers, and attaches `meta.parts` in [AggregateService.java](../src/main/java/dev/abramenka/aggregation/service/AggregateService.java#L59).
- Downstream clients: `AccountGroups`, `Accounts`, `Owners` are Spring HTTP service interfaces.
- Planning/execution: `AggregationPartPlanner`, `AggregationPartGraph`, and `AggregationPartExecutor` expand dependencies and run same-level parts concurrently.
- Enrichers: `AccountEnrichment`, `OwnersEnrichment`, `BeneficialOwnersEnrichment` are workflow-backed `AggregationPart`s.
- Error handling: `FacadeException` subclasses carry `ProblemDetail`; `AggregationErrorResponseAdvice` normalizes framework and unexpected errors.
- Validation flow: controller Bean Validation plus `ClientRequestContextFactory` manual query validation.
- Tests: controller, service, client, workflow, patch, error, actuator, and integration tests under `src/test/java`.

## 3. Key Findings

### Finding ID: ARCH-001

- Severity: **Critical**
- Confidence: **High**
- Area: API contract / errors / architecture
- Evidence:
  - `KeyedBindingStep` returns `SKIPPED / NO_KEYS_IN_MAIN` when no keys are extracted: [KeyedBindingStep.java](../src/main/java/dev/abramenka/aggregation/workflow/step/KeyedBindingStep.java#L115).
  - `RequireRootEntityOwnersStep` does the same for beneficial owners: [RequireRootEntityOwnersStep.java](../src/main/java/dev/abramenka/aggregation/enrichment/beneficialowners/RequireRootEntityOwnersStep.java#L40).
  - Tests lock this in: [AggregateServiceSelectionTest.java](../src/test/java/dev/abramenka/aggregation/service/AggregateServiceSelectionTest.java#L254).
  - Docs also declare data absence soft: [error-handling-design.md](error-handling-design.md#L13).
- Problem: Missing keys needed for enrichment are success-side outcomes.
- Why it matters: This directly contradicts the stated requirement that missing enrichment keys are errors.
- Recommendation: Add a root/main response contract validator per selected part. For required enrichers, map missing required keys to a fatal `MAIN-CONTRACT-VIOLATION` or a distinct facade-owned contract error.
- Effort: Medium
- Priority: Now

### Finding ID: ARCH-002

- Severity: **High**
- Confidence: **High**
- Area: downstream integration / errors
- Evidence:
  - `ResponseIndexer` silently returns an empty map for missing response paths or non-object items: [ResponseIndexer.java](../src/main/java/dev/abramenka/aggregation/workflow/binding/support/ResponseIndexer.java#L18).
  - `TargetMatcher` drops unmatched requested keys: [TargetMatcher.java](../src/main/java/dev/abramenka/aggregation/workflow/binding/support/TargetMatcher.java#L8).
  - `KeyedBindingStep` maps no matches to `DOWNSTREAM_EMPTY`: [KeyedBindingStep.java](../src/main/java/dev/abramenka/aggregation/workflow/step/KeyedBindingStep.java#L142).
  - Tests confirm partial enrichment is accepted: [AggregateServiceJoinTest.java](../src/test/java/dev/abramenka/aggregation/service/AggregateServiceJoinTest.java#L106).
- Problem: Required top-level enrichers can return incomplete or schema-invalid data without failing.
- Why it matters: Callers may receive a 200 response that looks aggregated but is missing required enrichment data.
- Recommendation: For required enrichers, validate response root shape, response item path, key fields, and requested-key coverage. Map violations to `ENRICH-CONTRACT-VIOLATION`.
- Effort: Medium
- Priority: Now

### Finding ID: ARCH-003

- Severity: **High**
- Confidence: **High**
- Area: API contract / security
- Evidence:
  - Public query parameter `fields` exists in `ClientRequestContext`: [ClientRequestContext.java](../src/main/java/dev/abramenka/aggregation/model/ClientRequestContext.java#L8).
  - It is parsed without a whitelist: [Projections.java](../src/main/java/dev/abramenka/aggregation/model/Projections.java#L19).
  - It is forwarded to the main service: [AggregateService.java](../src/main/java/dev/abramenka/aggregation/service/AggregateService.java#L76), [AccountGroups.java](../src/main/java/dev/abramenka/aggregation/client/AccountGroups.java#L20).
  - README documents only `detokenize` under query parameters: [README.md](../README.md#L224).
- Problem: Clients can change the main-service projection, potentially omitting required enrichment keys or exposing downstream fields.
- Why it matters: This can make required enrichment silently disappear and leaks downstream schema control into the public API.
- Recommendation: Remove `fields` from the public API, or whitelist it per endpoint/include and always force required enrichment fields.
- Effort: Medium
- Priority: Now

### Finding ID: ARCH-004

- Severity: **Medium**
- Confidence: **High**

### Finding ID: ARCH-005

- Severity: **Medium**
- Confidence: **Medium**
- Area: WebFlux / downstream integration
- Evidence:
  - All downstream methods are `POST`: [Accounts.java](../src/main/java/dev/abramenka/aggregation/client/Accounts.java#L16).
  - `DownstreamClientErrorFilter` retries transient request exceptions once for all clients: [DownstreamClientErrorFilter.java](../src/main/java/dev/abramenka/aggregation/client/DownstreamClientErrorFilter.java#L23).

| Validation error | 400 | 400 | `CLIENT-VALIDATION` | Good |

| Main object with empty/missing data | 200 with skipped parts | 502 | `MAIN-CONTRACT-VIOLATION` | Must change |

| Main 401/403 | 502 | 502 | `MAIN-AUTH-FAILED` | Good |
| Enrichment 401/403 | 502 | 502 | `ENRICH-AUTH-FAILED` | Good |
| Main timeout | 504 | 504 | `MAIN-TIMEOUT` | Good |

| Enrichment unavailable/503 | 504 | 504 or 502 | `ENRICH-UNAVAILABLE` | Existing choice is defensible |

## 5. WebFlux vs MVC Assessment

Recommendation: **Either is acceptable; do not keep WebFlux by default without measuring.**

WebFlux is implemented competently here: no obvious `.block()`, blocking I/O, or event-loop abuse was found. The downstream composition is naturally I/O-bound and same-level enrichment fanout benefits from reactive `WebClient`.

That said, Spring's own guidance says MVC is easier to write/debug, and WebFlux pays off mainly when non-blocking I/O plus latency/concurrency justify the complexity. This codebase has significant workflow complexity already. If expected traffic is moderate and the team is stronger with imperative Spring, MVC with `RestClient` or virtual-thread-friendly blocking clients would be simpler. If high concurrency and many downstream calls are expected, keep WebFlux.

## 6. Target Architecture Recommendation

Keep the current broad structure, but tighten contracts:

- `api`: controllers and public DTOs only.
- `application`: `AggregateService`, root validation, include planning.
- `part`: part graph, execution policy, result application.
- `enrichment.<name>`: business-specific part definitions and per-part required-key contracts.
- `integration.<dependency>` or `client`: HTTP service clients plus response contract validators.
- `error`: public problem catalog and mapper.

Recommended flow:
1. Validate request and include.
2. Compute required main fields from selected parts; ignore or constrain client `fields`.
3. Call main.
4. Validate main root shape and selected-part required keys.
5. Execute required enrichers.
6. Validate downstream body shape and requested-key coverage.
7. Merge results.
8. Return success only when required enrichment semantics are satisfied.

Testing layers:
- Controller contract tests.
- Application/service tests for required-key and main-empty cases.
- Client tests for every downstream status family.
- Contract tests for downstream schema violations.
- OpenAPI generation tests for key responses and headers.

## 7. Prioritized Remediation Plan

Phase 1: correctness and contract risks
1. Replace `NO_KEYS_IN_MAIN` soft success for required parts with fatal contract errors. Benefit: aligns with business requirement. Affects `KeyedBindingStep`, beneficial-owner precondition, service tests. Effort: Medium.
2. Add downstream response validators for top-level account/owners enrichers. Benefit: no partial 200s for required enrichment. Affects `ResponseIndexer`, `KeyedBindingStep`, enrichment tests. Effort: Medium.
3. Remove or whitelist public `fields`. Benefit: prevents clients from breaking enrichment. Affects `ClientRequestContextFactory`, `Projections`, README/OpenAPI/tests. Effort: Medium.

Phase 2: maintainability cleanup
1. Reconcile error catalog docs with code. Benefit: stable client contract. Affects `README.md`, `error-handling-design.md`, tests. Effort: Small.
2. Remove unused binding outcome model or wire it into execution. Benefit: clearer workflow model. Effort: Small.
3. Rename/publicly document odd response fields like `account1`/`owners1` if they are contractual. Benefit: clearer API. Effort: Medium.

Phase 3: observability/testing/build
1. Add OpenAPI components and generated-spec tests. Benefit: docs match runtime. Effort: Medium.
2. Add explicit tests for downstream 403, 503, 504, malformed schema, partial key coverage, and timeout integration. Benefit: locks error semantics. Effort: Medium.
3. Add downstream latency/error-code metrics or confirm Boot client observations are emitted with useful tags. Benefit: production diagnosis. Effort: Small.

## 8. Alternative Designs Considered

- Current design with contract fixes: best near-term option. It preserves the tested graph/workflow engine and fixes semantics.
- Cleaner facade/application-service design: better separation if `AggregateService` grows; move root validation and field planning into explicit collaborators.
- More domain-oriented design: useful if responses become stable business objects, but less suitable while downstream payloads are intentionally dynamic JSON.
- MVC-based design: simpler if concurrency is moderate and the team prefers imperative code. Loses some natural reactive fanout but reduces cognitive load.
- WebFlux-based design: appropriate if high concurrency and downstream latency dominate, provided required-contract validation is fixed.

## 9. Questions / Unknowns

- Are downstream POST endpoints guaranteed side-effect-free and idempotent?
- Is `fields` intended to be public API, internal-only, or accidental?
- Should enrichment 404 mean "no data" or a contract failure now that enrichment is required?
- Are `account1`, `owners1`, and `beneficialOwnersDetails` stable public fields?
- What traffic/concurrency level justifies WebFlux over MVC?

## 10. Final Recommendation

**Refactor before adding features.** Keep the existing architecture skeleton, but redesign the required-enrichment contract and downstream validation semantics now. The current implementation is internally coherent, but it implements an optional/soft enrichment model that conflicts with the stated business goal.
