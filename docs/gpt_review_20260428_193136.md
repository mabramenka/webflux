## 1. Executive summary

**Overall architecture rating: `Risky`.**

Not because the codebase is weak. The repository already has several mature architecture elements: centralized problem catalog, WebFlux orchestration, graph-based enrichment execution, metrics, CI, dependency verification, and a meaningful test suite.

The rating is `Risky` because the implementation does **not fully match the stated business requirement** that enrichment data is functionally required. The current behavior allows required-looking enrichment failures or missing keys to become HTTP 200 success responses with `meta.parts` outcomes such as `SKIPPED` or `EMPTY`.

### Top 3 strengths

1. **Strong centralized error model.**
   The repository has `ProblemCatalog`, `FacadeException`, `AggregationErrorResponseAdvice`, stable `errorCode`, `category`, `retryable`, `traceId`, `timestamp`, and dependency metadata.

2. **Good separation of runtime mechanics.**
   The controller is thin, downstream clients are isolated as HTTP service interfaces, orchestration is in `AggregateService`, graph/level execution is under `part`, and workflow mechanics are under `workflow`.

3. **Strong build and quality setup.**
   Java 21, Spring Boot 4.0.6, Gradle version catalog, NullAway, Error Prone, Spotless, JaCoCo threshold, OWASP Dependency Check, dependency verification, Boot 4 classpath guard, CI, and pinned GitHub Actions are present.

### Top 3 risks

1. **Required enrichment can currently complete as HTTP 200 with `meta.parts.SKIPPED/EMPTY`.**
   This conflicts with the project requirement that missing required enrichment keys should be treated as an error.

2. **The public API effectively returns a dynamic downstream-shaped `JsonNode`.**
   This is flexible, but weak as a public API contract. OpenAPI cannot reliably describe the response shape, and consumers may become coupled to downstream schemas.

3. **The workflow/path abstraction may be heavier than the current project needs.**
   For three enrichers, the repository already has graph planning, workflow execution, binding rules, JSON patching, and recursive traversal. This may be justified eventually, but it increases cognitive load and bug surface now.

### Fit for purpose

**Partially fit.**
The architecture skeleton is good, but error semantics must be corrected before adding more features.

## 3. Key findings

### Finding ID: ARCH-001

- **Severity:** Critical
- **Confidence:** High
- **Area:** API contract / errors / architecture
- **Evidence:**
  - `AggregateServiceSelectionTest.aggregate_softSkipsSelectedPartWhenMainPayloadHasNoRequiredKeys` expects success and `meta.parts.account.status = SKIPPED`, reason `NO_KEYS_IN_MAIN`.
  - The same test class expects downstream empty body and enrichment 404 to become success-side `EMPTY` outcomes.
  - `RequireRootEntityOwnersStep` returns `StepResult.skipped(NO_KEYS_IN_MAIN)` when root entity owners are absent.
- **Problem:**
  The project requirement says enrichment data is functionally required and missing required keys in main response should be an error. Current implementation treats this as a successful aggregation with metadata.
- **Why it matters:**
  API consumers can receive HTTP 200 with incomplete business data. They must inspect `meta.parts` to discover a failure-like condition. That is dangerous for required enrichment.
- **Recommendation:**
  Split outcome semantics:
  - required selected/default enrichers: missing required keys, downstream empty, downstream 404, invalid enrichment contract => fail request;
  - optional enrichers only: allow `EMPTY/SKIPPED/FAILED` in `meta.parts`.
  Add explicit tests proving required enrichers fail.
- **Effort:** Medium
- **Priority:** Now

### Finding ID: API-001

- **Severity:** High
- **Confidence:** High
- **Area:** API contract / maintainability
- **Evidence:**
  - `AggregateController` returns `Mono<JsonNode>`.
  - README states the service keeps payloads dynamic via `JsonNode` / `ObjectNode`.
  - `AggregateService.attachMeta` mutates downstream-shaped root by adding `meta.parts`.
- **Problem:**
  The public response contract is effectively “whatever main downstream returned, plus fields added by the facade.” This is flexible but weak as a public API.
- **Why it matters:**
  OpenAPI will not describe the real response shape well. Clients couple to downstream schemas. Any downstream shape drift becomes public facade drift.
- **Recommendation:**
  Either define a stable public envelope such as `data`, `included`, `meta`, `errors`, or keep dynamic `data` but publish an explicit JSON Schema / OpenAPI schema for required fields and extension points.
- **Effort:** Medium/Large
- **Priority:** Now

### Finding ID: INT-001

- **Severity:** High
- **Confidence:** Medium
- **Area:** downstream integration / correctness
- **Evidence:**
  `OwnersEnrichment` extracts owner keys from both `basicDetails.owners[*].id` and `basicDetails.owners[*].number`, and indexes response by `individual.number` and `id`, but the write rule matches only `basicDetails.owners[*].id` to `individual.number`.
- **Problem:**
  The fallback key strategy appears inconsistent. If main payload has only `number`, the client may fetch owners but fail to attach them.
- **Why it matters:**
  This can produce incomplete enrichment while still looking successful.
- **Recommendation:**
  Add a regression test where main payload has only `basicDetails.owners[*].number`. If it currently fails to attach, change `WriteRule.MatchBy` to support the same fallback set used during extraction/indexing.
- **Effort:** Small/Medium
- **Priority:** Now

### Finding ID: ERR-001

- **Severity:** Medium
- **Confidence:** High
- **Area:** errors / API contract
- **Evidence:**
  `ProblemCatalog.MAIN_UNAVAILABLE` and `ENRICH_UNAVAILABLE` both use `HttpStatus.GATEWAY_TIMEOUT`, the same status family used for timeouts. `DownstreamClientException` maps downstream `503` to `*_UNAVAILABLE`.
- **Problem:**
  Timeout and unavailable are semantically different but both become HTTP 504.
- **Why it matters:**
  API consumers and monitoring cannot distinguish “dependency timed out” from “dependency returned unavailable” using HTTP status.
- **Recommendation:**
  Keep distinct `errorCode`s, but map unavailable to either:
  - `502 Bad Gateway` when upstream returned a bad dependency status; or
  - `503 Service Unavailable` if the facade wants to communicate temporary service unavailability.
  Reserve `504` for real timeout.
- **Effort:** Small
- **Priority:** Soon

### Finding ID: WEBFLUX-001

- **Severity:** Medium
- **Confidence:** High
- **Area:** WebFlux / resilience
- **Evidence:**
  `DownstreamClientErrorFilter` applies one retry with 200ms backoff only for `WebClientRequestException` and `TimeoutException`. It maps HTTP error statuses after retry, so downstream `503/504` statuses are not retried. It also retries POST requests without visible idempotency policy.
- **Problem:**
  Retry policy is global, hardcoded, and incomplete relative to the public `retryable` semantics.
- **Why it matters:**
  Required downstream POST calls may be duplicated if they are not idempotent. Conversely, retryable HTTP statuses are not actually retried.
- **Recommendation:**
  Move retry policy into per-client config. Retry only where the downstream operation is known idempotent or has an idempotency key. Decide explicitly whether HTTP `503`, `504`, and `429` should be retried.
- **Effort:** Medium
- **Priority:** Soon

## 5. WebFlux vs MVC assessment

**Recommendation: either is acceptable, but do not keep WebFlux just because it is already there.**

Given the actual code, **keeping WebFlux is defensible**:

- downstream calls are remote I/O;
- same-level enrichers can run concurrently;
- clients are WebClient-backed HTTP service clients;
- current chain is mostly non-blocking and returns `Mono`;
- no obvious blocking JDBC/JPA/filesystem calls in request path were observed.

Spring’s own documentation describes WebFlux as a non-blocking stack designed for concurrency with a small number of threads, but also states that if Spring MVC works fine, there is no need to switch. Imperative programming is easier to write, understand, and debug. WebFlux benefits are strongest when there is latency and unpredictable network I/O.

### Keep WebFlux if

- expected concurrency is high;
- downstream latency dominates;
- team is comfortable with Reactor;
- no blocking libraries enter request path;
- observability and tests cover reactive behavior.

### Switch to MVC if

- traffic is moderate;
- team maintainability matters more than thread efficiency;
- most developers struggle with Reactor;
- future logic becomes mostly imperative mapping/validation;
- blocking dependencies appear.

### Independent verdict

**Keep WebFlux for now, but only after fixing contract semantics.**
Switching to MVC would not solve the current biggest issue. The biggest issue is not reactive vs servlet; it is that required enrichment failures are modeled as success.

### Main components

- **Application/facade service:** orchestration policy and flow.
- **Enrichment parts:** business-specific logic with explicit required keys and response validators.
- **Integration clients:** HTTP details, timeouts, retries, header policy.
- **Error layer:** public error contract only; no raw downstream leakage.

### Error model

Keep Problem Details. Current direction is good.

Change semantics:

```text
Required enrichment missing required keys -> error
Required enrichment downstream empty/404 -> error
Optional enrichment missing/empty/404 -> meta.parts outcome
```

### Validation model

- Bean Validation for external input.
- Service-layer invariants for things external input cannot guarantee:
  - main response object shape;
  - main response contains required enrichment keys;
  - downstream response has expected `data` shape;
  - key matching produced required attachments.

### Downstream abstraction

Each downstream client should have:

```text
base URL
connect/read timeout
retry policy
auth/header forwarding policy
response validator
error mapper
```

Retry policy should be per client and per operation, not one global hardcoded filter.

### Aggregation flow

```text
1. Validate request.
2. Parse include.
3. Build effective plan.
4. Call main dependency.
5. Validate main response object and required keys.
6. Execute same-level enrichers concurrently.
7. Validate each downstream response.
8. Merge results.
9. Return stable public response.
```

### Testing layers

- Unit tests: planner, error mapper, key extraction, merge rules.
- API tests: validation and ProblemDetail shape.
- Client tests: headers, query params, statuses, malformed JSON, timeouts.
- Contract tests: all required error scenarios.
- Regression tests: owners fallback number-only case.
- Observability tests: traceparent forwarded downstream.

## 8. Alternative designs considered

### Alternative A — Current design with fixes

**Description:** Keep WebFlux, graph planner, workflow executor, Problem Details, dynamic JSON; fix required-enrichment semantics and tests.

**Better because:**

- least disruptive;
- preserves existing investment;
- current test/build infrastructure remains useful.

**Worse because:**

- dynamic public contract remains weak;
- workflow DSL remains complex.

**Verdict:** Best short-term option.

---

### Alternative B — Cleaner facade/application-service design

**Description:** Keep WebFlux clients, but move business policy into a clearer `AggregationFacade` and simplify enrichers to ordinary Java components.

**Better because:**

- easier to reason about required vs optional;
- less DSL magic;
- better for team onboarding.

**Worse because:**

- requires medium refactor;
- may duplicate some generic workflow behavior.

**Verdict:** Best medium-term target.

---

### Alternative C — More domain-oriented design

**Description:** Convert main/downstream JSON into internal domain models, aggregate typed objects, then map to public DTO.

**Better because:**

- strongest contract;
- best validation;
- best OpenAPI;
- easier tests for business invariants.

**Worse because:**

- more mapping code;
- less flexible if downstream schemas are unstable;
- may be overkill if facade must stay schema-light.

**Verdict:** Good if API consumers need a stable long-lived contract.

---

### Alternative D — MVC-based design

**Description:** Replace WebFlux controller stack with Spring MVC, keep WebClient or use blocking clients.

**Better because:**

- easier debugging and onboarding;
- simpler mental model;
- often enough for moderate traffic.

**Worse because:**

- loses natural non-blocking concurrency model;
- parallel downstream composition becomes less elegant;
- switch cost does not address current contract bug.

**Verdict:** Acceptable only if team maintainability outweighs I/O scalability.

## 9. Questions / unknowns

These do not block the review, but they materially affect final design:

1. Are `account`, `owners`, and `beneficialOwners` always required when selected, or only required for some product contexts?
2. Is `include = null` really “all required enrichers”, or should it be “default useful enrichers”?
3. Should enrichment `404` mean “not found data” or “dependency contract failure”?
4. Are downstream POST calls idempotent?
5. Is the facade expected to expose the main downstream schema as-is, or should it own a stable public response?
6. Are downstream services expected to understand W3C `traceparent`?
7. Is `fields` a public query parameter that callers may control, or internal projection logic?
8. Should downstream unavailable be exposed as `502` or `503`?

---

## 10. Final recommendation

**Do not redesign the whole project now. Do refactor before adding features.**

Clear verdict:

```text
Continue with the current architecture skeleton,
but fix required-enrichment semantics immediately.
```

The codebase has a solid foundation: thin controller, centralized Problem Details, HTTP service clients, graph execution, and strong build gates. But the current behavior is unsafe for the stated business goal because required enrichment can silently degrade into HTTP 200 with `meta.parts`.

Recommended order:

1. Fix contract semantics first.
2. Rewrite tests that currently preserve soft-success behavior.
3. Fix owner fallback matching.
4. Clarify response contract/OpenAPI.
5. Only then continue adding enrichers or expanding workflow DSL.

Technology verdict:

```text
Keep WebFlux for now.
Do not switch to MVC unless team maintainability or blocking dependencies become the dominant concern.
```

The main architectural problem is not WebFlux. The main architectural problem is that the service currently mixes **required business data** with **optional partial-success mechanics**.
